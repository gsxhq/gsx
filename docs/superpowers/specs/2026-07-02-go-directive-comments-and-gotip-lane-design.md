# Go directive comment pass-through + go1.27-tip test lane

Date: 2026-07-02
Status: approved

Two independent slices that came out of the generic-components review.

## Slice 1: pass program-significant comments through to the generated `.x.go`

### Problem

A `.gsx` file's pre-package comment block is captured by the parser
(`File.Doc`, parser/file.go) and preserved by the formatter, but codegen drops
it: `emit.go` writes only the generated-code marker and the package clause. A
`//go:build` constraint written in a `.gsx` therefore silently vanishes from
the generated `.x.go`, which then builds on every platform regardless of tags.

### Rule

From the pre-package comment block, copy **comments that are significant to
the program** Рђћ lines the Go toolchain acts on Рђћ verbatim into the generated
file. Everything else (license headers, doc prose) stays `.gsx`-only.

Significant lines are:

- `//go:<directive>` Рђћ any Go directive comment (`//go:build`, `//go:generate`,
  `//go:debug`, Рђд). Directives have no space after `//` (a `// go:build` line
  is prose, per the toolchain's own rules).
- `// +build <expr>` Рђћ the legacy pre-1.17 constraint spelling. Copied because
  gofmt keeps it paired with `//go:build` and older toolchains only read this
  form.

Deliberately **not** copied: `//line` directives (they would corrupt the
`//line` mapping codegen itself emits) and all non-directive comments.

### Emission

Directive lines are emitted between the generated-code marker and the package
clause, preserving their relative order, with a blank line before `package`:

```go
// Code generated by gsx; DO NOT EDIT.

//go:build !windows
//go:generate some-tool

package views
```

This placement satisfies both the generated-code-marker convention and the
constraint-placement rules (`//go:build` before the package clause, blank line
separating it from the clause).

### Semantics

- **Generation is build-context-independent.** `gsx generate` emits a `.x.go`
  for every `.gsx` regardless of host GOOS/GOARCH/tags. Constraints take
  effect at `go build`, so one generate pass serves cross-compilation.
- **Known limitation (documented, not solved):** two `.gsx` files with
  mutually exclusive constraints declaring the same component name fail
  analysis with a duplicate declaration, because module analysis type-checks
  all `.gsx` files in a package as one unit and does not tag-filter. Distinct
  names across platform files work. Recorded in ROADMAP; per-build-config
  analysis is out of scope here.
- Generated `.x.go` filenames never acquire *implicit* filename constraints
  (`views_windows.gsx` Рєњ `views_windows.x.go`, whose base `views_windows.x`
  does not match Go's `*_GOOS` filename rule), so the explicit `//go:build`
  comment is the one and only constraint mechanism Рђћ document this.
- Cache correctness: the incremental cache keys on full source content, so a
  directive edit changes the key; no `computeKey` change needed.

### Tests / docs / siblings

- Corpus case pinning `input.gsx` (with a host-neutral constraint such as
  `//go:build !never` plus a second directive) Рєњ `generated.x.go.golden` +
  `render.golden`, so parse Рєњ generate Рєњ render all run everywhere.
- Unit test that a tag-excluded generated file is genuinely dropped by
  `go build` (probe module with a constraint that excludes the host).
- Negative case: prose comments and `//line` in the doc block do NOT appear in
  the generated file.
- Docs: `docs/guide/` section on build constraints in `.gsx`.
- No sibling-repo syntax work: these are ordinary comments already handled by
  tree-sitter/vscode/CodeMirror highlighting.

## Slice 2: actually run the go1.27 generic-methods test

### Problem

`TestGenericMethodComponentGo127` skips unless the active toolchain's
`go/parser` accepts generic methods. Local dev and CI both pin go1.26.1, so
the test has never executed anywhere.

### Design

- **Make target `test-gotip`**: requires `gotip` on PATH; if missing, fail
  with the install hint (`go install golang.org/dl/gotip@latest && gotip
  download`). Runs `GSX_REQUIRE_GENERIC_METHODS=1 gotip test
  ./internal/codegen -run Go127 -count=1 -v`.
- **Skip Рєњ failure guard**: when `GSX_REQUIRE_GENERIC_METHODS=1` is set, the
  test fails (instead of skipping) if `supportsGenericMethods()` is false.
  Without the guard the tip lane would green-light forever while testing
  nothing Рђћ which is the current state on 1.26.1. (Test-only env var, not
  user config; not registered in `envOverrides`.)
- **CI job `go-tip`** in ci.yml: installs gotip the same way, runs the same
  target, `continue-on-error: true` so tip breakage never blocks merges. Main
  lanes stay pinned to `GO_VERSION`.

### Follow-up trigger

When go1.27 is released, bump `GO_VERSION` in ci.yml; the gated test then runs
in the main lanes and the tip lane keeps covering the next dev cycle.
