# Runtime Render Performance Audit

**Date:** 2026-07-21  
**Status:** Approved for planning  
**Repositories:** `gsx`, sibling `gsx-bench`

## 1. Goal

Improve the rendering performance experienced by GSX users by auditing the
complete generated render path: generated `.x.go` code, the root runtime
package, and the boundary between them.

The audit is evidence-gated. It does not assume that a runtime rewrite is
needed, and it does not preserve an inferior API merely for pre-alpha
compatibility. Small changes must produce a repeatable improvement in the path
they target. Large generated/runtime ABI changes require a material improvement
in representative end-to-end rendering, not only a microbenchmark win.

`~/work/one-learning` is not a GSX consumer and is explicitly outside this
audit.

## 2. Current State

The core `gsx` worktree starts from a passing `make check` baseline.

The sibling `gsx-bench` repository is stale in two distinct ways:

1. Its committed generated files call an older `Writer.Spread` signature and
   no longer compile against current GSX.
2. Its authored `.gsx` files rely on the removed implicit `attrs` binding and
   cannot be regenerated by current GSX without a source migration.

Existing core microbenchmarks show that the empty component-root path is
already effectively free, while a small non-empty forwarding bag performs
meaningful work and allocates. This makes non-empty attribute forwarding,
class/style composition, spread classification, and generated call shape the
first profiling area. It is a lead, not a preselected fix.

A previous experiment established that merely replacing closure-backed nodes
with struct-backed nodes does not remove per-child allocations at the
`Writer.Node(Node)` interface boundary. The audit will not repeat that design.
A genuinely different direct-render ABI may be prototyped only if refreshed
profiles show composition remains a dominant user-facing cost.

## 3. Design Principles

- Measure absolute GSX performance first. Comparisons with templ and
  `html/template` provide context, not the acceptance criterion.
- Keep security semantics exact. HTML, URL, srcset, JavaScript, and CSS handling
  must retain their current tested behavior; no approximate fast path is
  acceptable.
- Optimise the generated code and runtime as one system. Package boundaries do
  not define the audit boundary.
- Preserve fast paths that are already good. A new general path must not impose
  work on static markup, empty bags, allocation-flat loops, numeric writes, or
  allocation-free escaping.
- Prefer a simpler final ABI when measurements justify a change. Do not add
  compatibility wrappers, dual paths, or deprecated shims in this pre-alpha
  stage.
- Change one performance variable at a time so benchmark attribution remains
  credible.

## 4. Benchmark Architecture

### 4.1 Core repository

The core repository owns focused microbenchmarks close to the implementation.
They isolate runtime primitives and exact generated call shapes, including:

- writer and escaping sinks;
- non-empty `Attrs` lookup, class/style aggregation, and spread rendering;
- conditional/folded attribute composition;
- component composition and `Node` dispatch;
- generated static, dynamic, numeric, URL, and child writes.

Microbenchmarks diagnose a cost and guard a retained optimisation. They are not
alone sufficient evidence for a broad ABI change.

### 4.2 `gsx-bench` repository

`gsx-bench` owns representative external workloads. Its authored GSX and templ
sources are the source of truth; generated Go files remain committed and must
be regenerated from the paired worktrees.

The refresh will retain output-equivalence tests and cover at least:

- a mostly static document with dynamic text and URL attributes;
- an allocation-flat loop;
- repeated child-component composition;
- a realistic whole page;
- escaping-heavy text;
- numeric interpolation;
- a custom class merger;
- non-empty forwarded attributes with scalar, boolean, class, style, and URL
  values;
- conditional/folded attributes;
- parallel rendering;
- pooled-buffer and discard destinations, with cold builder results retained
  only as context.

Scenario sizes and input values remain stable within an audit so before/after
runs compare identical work. Shared GSX/templ scenarios must render equivalent
HTML modulo explicitly documented browser-irrelevant differences.

### 4.3 Measurement protocol

Performance comparisons use the pinned project Go version and the same machine,
power state, benchmark inputs, destination, `GOMAXPROCS`, and generator outputs.
Each comparison records multiple interleaved samples suitable for `benchstat`,
including `ns/op`, `B/op`, and `allocs/op`.

CPU profiles, allocation profiles, compiler escape diagnostics, generated-code
inspection, and `gopls references` identify why a result moved. Profiles are
diagnostic evidence; repeated benchmark comparisons decide whether a change is
kept.

Raw profiles and temporary benchmark outputs stay outside the repositories.
The final findings, commands, environment, retained changes, and rejected
experiments are recorded in
`docs/superpowers/notes/2026-07-21-runtime-render-performance-audit.md`.

## 5. Audit and Experiment Flow

### Phase A: Restore the measurement system

Migrate `gsx-bench` to current authored syntax, regenerate GSX output from the
paired core worktree, verify templ output is current, and restore all tests.
Add missing scenarios before changing the runtime so they establish an honest
baseline.

### Phase B: Establish and explain the baseline

Run repeated external and core benchmarks. Profile the representative GSX
scenarios and inspect escape decisions and generated call sites. Produce a
ranked hotspot inventory that distinguishes:

- required destination cost from GSX overhead;
- one-time per-render cost from per-element or per-child cost;
- unavoidable user computation from runtime/codegen cost;
- heap allocations from stack work;
- benchmark artifacts from production-relevant paths.

### Phase C: Run focused experiments

Candidate families are investigated in measured order rather than assumed
priority:

1. Attribute forwarding: repeated bag scans, last-wins handling, URL-name
   classification, class/style extraction, and spread writes.
2. Generated call shape: repeated construction/passing of static metadata,
   avoidable helper calls, and opportunities to coalesce work without changing
   semantics.
3. Writer and escaping sinks when profiles show meaningful time there.
4. Conditional/folded attribute materialisation and temporary bags.
5. Component composition. A direct-render ABI experiment is allowed only after
   the smaller measured paths and only if composition remains dominant.

Each experiment begins with a benchmark or corpus case that exposes the exact
cost or semantic contract. Only one hypothesis changes per comparison.

### Phase D: Keep, reject, and consolidate

A retained change receives correctness tests, corpus coverage for generated
shape, regenerated goldens, focused benchmarks, and documentation. A rejected
experiment is removed cleanly and its result is documented so the repository
does not revisit it without new evidence.

After retained experiments, rerun the complete external suite to detect
interactions and consolidate the final ABI. Because compatibility is not a
goal, superseded helpers are deleted rather than wrapped.

## 6. Acceptance Gates

All performance claims require repeated before/after samples. A result is
credible when `benchstat` reports a statistically significant change and the
effect is larger than observed run-to-run noise.

For a focused, low-complexity change, a useful time reduction or removal of a
heap allocation on a repeated hot path can justify landing it, provided no
representative scenario regresses materially.

For a large ABI or codegen redesign, the bar is deliberately higher:

- a clear double-digit end-to-end time improvement in an affected
  representative workload, or a substantial allocation reduction that matters
  under realistic repetition;
- no unexplained material regression in other representative workloads;
- a simpler or equally coherent generated/runtime model after obsolete paths
  are removed;
- full correctness, security, corpus, and downstream benchmark verification.

If measurements are mixed, below noise, or limited to an artificial
microbenchmark, the large change is rejected.

## 7. Correctness and Safety

Every retained runtime or generated-code change must preserve:

- byte output for existing corpus cases and benchmark equivalence fixtures,
  except for an explicitly reviewed semantic correction;
- first-error behavior and no-op-after-error writer semantics;
- attribute order and duplicate resolution;
- boolean-attribute behavior;
- class/style caller precedence and configured merger call counts;
- URL, image URL, srcset, refresh, JavaScript, CSS, and HTML escaping contracts;
- nil-node and child ordering behavior;
- concurrent rendering without shared mutable render state.

Security-sensitive fast paths must be proven from existing exact classifiers or
compile-time knowledge. No heuristic replacement is permitted.

## 8. Verification

During development:

- targeted root tests and benchmarks;
- semantic corpus regeneration followed by a non-update verification run;
- formatter corpus regeneration when generated source formatting changes;
- `gopls check -severity=hint` on changed Go files;
- `go test -race` for affected concurrent/runtime packages where relevant;
- `make check` as the inner gate.

Before completion:

- `make ci` and `make lint` in `gsx`;
- `make generate`, output-equivalence tests, `go test -count=1 ./...`, and the
  full repeated benchmark suite in `gsx-bench`;
- generated-file drift checks in both repositories;
- fresh final profiles for materially changed hotspots;
- review and update of `docs/ROADMAP.md` and
  `docs/guide/performance.md` when retained work changes their claims;
- one independent adversarial review using throwaway probe programs against
  the final generated/runtime ABI.

## 9. Deliverables

- A current, reproducible `gsx-bench` repository with representative coverage.
- A written audit report with baseline, hotspot attribution, experiments,
  rejected ideas, and final measurements.
- Only those runtime/codegen optimisations that pass the evidence gates.
- Tests and corpus goldens that pin every retained semantic/generated shape.
- Updated canonical performance and roadmap documentation where warranted.
- Separate focused commits for benchmark restoration, each retained
  optimisation, and final documentation so measurements remain attributable.

Publishing, pull requests, merging, or website synchronization are outside this
design unless requested separately.
