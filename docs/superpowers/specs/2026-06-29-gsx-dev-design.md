# `gsx dev` — gsx owns the dev loop

**Status:** approved design (brainstormed 2026-06-29), ready for an implementation plan.

**One-line:** Add a `gsx dev` command that owns the whole dev feedback loop in one warm
Go process — generate (warm `codegen.Module`) → build → run the Go server → drive the
browser — replacing the scaffold's Taskfile + wgo + tee glue, and flipping `.gsx`
generation from a vite-plugin subprocess into gsx dev's in-process warm core.

## 1. Goal & non-goals

### Goal
`gsx dev` runs a single foreground process that watches the project, regenerates `.x.go`
through a warm `*codegen.Module`, rebuilds and restarts the Go server on `.go`/`.env`
changes, and tells the browser what happened (error overlay on failure, full-reload on
success) — all with clean process-group teardown on Ctrl-C. It spawns and supervises vite
(the front door) and the Go server as children. The scaffold drops to one dev command
(`go tool gsx dev`) and a handful of obvious files: no Taskfile, no wgo, no `tee`, no
daemon-spawning vite config.

This is validated: a stdlib-only prototype (`devspike`) demonstrated clean teardown
(4/4 faithful PTY Ctrl-C), selective server restart with vite untouched and no orphan
accumulation, and native server-log capture — with no external task runner.

### Why now (the motivating realization)
`.gsx → .x.go` generation lived in the vite plugin (it spawns `gsx generate --watch` and
reads its NDJSON) for exactly one reason: **only the plugin could turn a codegen error
into a browser overlay**, because the plugin was the only persistent process wired to the
browser. Once `gsx dev` exists, gsx dev *is* that persistent process. The original reason
evaporates, so generation moves to where the rest of the loop already is.

### Non-goals
- **Replacing vite or Task generally.** `gsx dev` does exactly one thing well — the gsx
  feedback loop — and stays a well-behaved foreground process so a Makefile / Taskfile /
  process-compose can compose it with everything else (db, tailwind, workers). It is not a
  general task runner.
- **Windows in v1.** Teardown uses unix/macOS process groups (`Setpgid` + `kill(-pgid)`).
  Windows (job objects / `CREATE_NEW_PROCESS_GROUP`) is a documented follow-up.
- **`--watch`/`--exclude` flags in v1.** Watch/exclude sets are fixed sensible defaults;
  flags get added only when a real need appears (YAGNI).
- **Changing generated bytes or the warm core's semantics.** `gsx dev` reuses the existing
  `gen/watchsession.go` + `gen/watch.go` warm watch path as-is; it does not alter codegen.

## 2. Background — today's dev loop

Three actors, glued by Task:
- **vite plugin** (`vite-plugin-gsx/src/index.ts`) spawns `gsx generate --watch` as a
  daemon, reads its NDJSON (`event: "generated"`, `ok`), and drives the browser: `ok:false`
  → Vite error overlay (error text comes from the event payload, not a file); `ok:true`
  after an error → recovery full-reload. It also exposes `/__reload` (POST → broadcast
  full-reload) and `devFallback` (`/__dev/status` → `{up, log}`, tailing `tmp/dev.log`;
  plus the server-down interstitial).
- **wgo** watches `.go` and rebuilds/restarts the Go server.
- **Task** (`Taskfile.yml`) runs vite + server in parallel, tees server output to
  `tmp/dev.log`, and (failing to) tear down on Ctrl-C — the pain that started this work.

The Go server (`main.go.tmpl`) calls `vite.NotifyReload(devURL)` after `ListenAndServe`
so the browser reloads once the server is back up (best-effort POST with retries to cover
the cold-start race).

Relevant existing infrastructure we build on:
- `gen/main.go` dispatcher: subcommands via `switch rest[0]` (`generate`, `fmt`, `init`,
  `lsp`, …). New command = a `case "dev"` → `runDev(args, stdout, stderr, cfg, workDir)`.
- `gen/watch.go` / `gen/watchsession.go`: fsnotify watcher (100ms debounce; watches
  `.gsx`/`.go`/`go.mod`/`go.sum`; excludes `tmp`/`dist`/`node_modules`/`.git`) driving one
  warm `*codegen.Module` per module root, with incremental `regenDir` + reverse-closure.
- `fsnotify` and `golang.org/x/tools` already in `go.mod` (single module
  `github.com/gsxhq/gsx`).
- `gen/configfile.go` (`gsx.toml`, strict decode), `gen/envconfig.go`, `gen/options.go`,
  resolved codegen `config` struct (`gen/main.go`).

## 3. Architecture

One warm process owns the loop. `gsx dev` reuses the warm watch session and supervises two
children (vite + the Go server). Each settled file-change cycle runs in order:

1. **Regenerate** the changed dirs through the warm `codegen.Module` (in-process — no
   `gsx generate` subprocess).
2. **Rebuild + swap the Go server** if any `.go` changed (source *or* freshly-generated
   `.x.go`). Generation and server-rebuild are two stages of one cycle, so fresh `.x.go`
   provably exists before the server build runs.
3. **Drive the browser** — POST to the vite plugin: failure → overlay (with error text in
   the payload); success/recovery → full-reload.

```
                 ┌──────────────────────── gsx dev (one process) ─────────────────────┐
   .gsx/.go ───▶ │ fsnotify ─▶ warm Module.regen ─▶ [.go changed?] ─▶ build-then-swap  │
   .env     ───▶ │              (reload env, restart server, no rebuild)               │
                 │ children:  vite (npm/npx, persistent)   Go server (build/run/watch) │
                 │ outputs:   prefixed terminal   +   tmp/dev.log (codegen+build+srv)   │
                 │ browser:   POST overlay/reload ─────────────────────────────────────┼──▶ vite plugin
                 │ teardown:  SIGINT ─▶ kill(-pgid) SIGTERM→SIGKILL each child          │     (gsx({watch:false}))
                 └─────────────────────────────────────────────────────────────────────┘
```

### 3.1 The browser-feedback bridge (the one cross-repo change)
Today the plugin *spawns* `gsx generate --watch` and reads its NDJSON. We flip the
direction: the plugin gains an option — `gsx({ watch: false })` — that, instead of spawning
a daemon, exposes an HTTP endpoint to **receive** the same event shape from gsx dev. The
plugin's overlay/reload logic is unchanged; only the event *source* flips from
subprocess-stdout to HTTP POST.

- **Overlay is purely event-driven, in-process.** Error text travels in the POST payload;
  the plugin renders Vite's overlay from it directly, never reading a file. So the normal
  failure UX (app still serving the last-good build + overlay showing exactly what broke)
  works **with or without** `tmp/dev.log`.
- `/__reload` and the `tmp/dev.log` → `devFallback` interstitial wiring stay exactly as
  they are.
- Cold-start race (gsx dev generates/POSTs before vite is up) is handled the same way
  `NotifyReload` does today: best-effort POST with short retries.

This work spans two repos: `gsx` (the `gsx dev` command) and `vite-plugin-gsx` (the
`watch:false` receive-events option).

### 3.2 Process lifecycle & error handling — build-then-swap
The key refinement over the prototype: **a broken build never takes the app down.**

- **Codegen fails** → diagnostics to `tmp/dev.log` + terminal; POST overlay (error text in
  payload); **keep the last-good server running** (don't rebuild against stale `.x.go`).
- **Build fails** → `go build` does not replace the binary on failure, so the running
  server stays up; error to `dev.log` + overlay. Fix it → next cycle swaps in the new
  binary.
- **Build succeeds** → kill old server (process-group), start new; poll `GO_PORT` until it
  accepts connections, then POST full-reload.
- **Server crashes at runtime** → capture exit to `dev.log`; the proxy gets ECONNREFUSED →
  `devFallback` interstitial shows it's down (tailing `dev.log`); next successful change
  restarts it.
- **`.env*` changes** → reload env, **restart the server with fresh env (no rebuild)**,
  re-poll readiness, POST reload. Vite handles its own `.env` reload natively (its
  `vite.config.ts` reads the file via `loadEnv` and Vite watches `.env*`), so gsx dev only
  restarts the process *it* owns.
- **Vite dies** → reported (not silently swallowed); v1 does not auto-restart it.

Teardown (validated): each child is started in its own process group (`Setpgid: true`); on
SIGINT/SIGTERM gsx dev signals the group (`kill(-pgid)`) SIGTERM→SIGKILL after a timeout,
and waits. No backgrounding — gsx dev is a clean foreground process so orchestrators can
compose and signal it.

### 3.3 Reload timing moves out of the user's server
Because gsx dev knows when it (re)starts the server and can poll `GO_PORT` for readiness,
**`main.go` drops the `vite.NotifyReload` dev hook** and becomes a plain Go HTTP server
(still using `vite.New()` for asset URLs + graceful shutdown). The one piece of dev-tooling
plumbing leaves the user's server file; gsx dev owns reload timing (start → readiness-poll →
POST reload). Port readiness is a TCP-accept check (the same guarantee `NotifyReload`'s
retry gave), not a heuristic.

### 3.4 The watch → action map

| Changed | Action |
|---|---|
| `.gsx` | warm-regenerate → if `.x.go` changed, build-then-swap server → reload |
| `.go` (source) | warm-regenerate reverse-closure → build-then-swap → reload |
| `go.mod` / `go.sum` | reopen warm module (full) → build-then-swap → reload |
| `.env*` (`.env`, `.env.local`, `.env.development`) | reload env → restart server (no rebuild) → reload |

Excludes: `tmp/`, `dist/`, `node_modules/`, `.git/`.

### 3.5 The unified `tmp/dev.log`
Now that gsx dev owns all three backend stages, `tmp/dev.log` is promoted from the
prototype's server-only capture to the **unified backend-status log**: codegen diagnostics +
`go build` errors + server stdout/stderr, written via `io.MultiWriter(terminal, logf)`,
**truncated once at startup, appended per cycle** with a timestamped separator (recent
history survives restarts; the interstitial just tails the end). It serves two secondary
purposes — the server-down interstitial's "why" display, and frictionless agent grep
(`grep tmp/dev.log`). With `--no-log` both degrade gracefully; the overlay (§3.1) still
works fully because it's event-driven, not file-driven. The terminal stays the human's
primary view (prefixed combined output, including vite).

## 4. Configuration — clean default + `gsx.toml [dev]` + flag overrides

Resolution (mirrors the project's option > env > config layering, minus env — dev config is
inherently dev-only, so no `GSX_*` layer): **CLI flag → `gsx.toml [dev]` → convention
default.**

| Knob | Convention default | `gsx.toml [dev]` | CLI flag |
|---|---|---|---|
| project dir | cwd | — | `gsx dev [dir]` |
| web front-door | `npx vite` | `web = ["npx","vite"]` | `--web <cmd>` / `--no-web` |
| server build | `go build -o tmp/app .` | `build = ["go","build","-o","tmp/app","."]` | `--build <cmd>` |
| server run | `tmp/app` | `run = ["tmp/app"]` | `--run <cmd>` |
| backend log | `tmp/dev.log` | `log = "tmp/dev.log"` | `--log <path>` / `--no-log` |

- **`--no-web`** is the composability lever: gsx dev manages only the Go side (generate +
  build + server + browser-notify via `VITE_DEV_URL`) while the user runs vite however they
  like in another pane/task.
- **Command form is argv.** `gsx.toml [dev]` uses **array form** for exactness; CLI flags
  are whitespace-split for the common case (no hidden quote-parsing heuristic — array when
  you need precision, flag when it's simple).
- **`npx vite`, not `npm run dev`,** is the web default: the scaffold maps `npm run dev` →
  `go tool gsx dev`, so a `npm run dev` default would recurse. gsx dev execs argv directly
  (no shell — honors the "no `sh -c`" rule).
- **Ports stay in `.env`,** propagated to children; gsx dev binds nothing itself (it only
  *reads* `GO_PORT` for the readiness poll and `VITE_DEV_URL` for the event POST).

**Consistency with the config rules (`CLAUDE.md`):** the `[dev]` table is read **only by
`runDev`** — a separate `devConfig`, *not* part of the codegen `config` struct and **not**
folded into `computeKey`, because dev knobs don't change generated output. It sits beside
the codegen config in `gsx.toml` without touching the incremental cache.

## 5. Scaffold migration (`gen/templates/init/simple`)

**Deleted:** `Taskfile.yml` (the task-runner concept disappears); `wgo` removed from the
template `go.mod` (`tool` + `require`).

**Simplified:**
- `package.json` → `"dev": "go tool gsx dev"`, `"build": "vite build"`; deps just `vite` +
  `@gsxhq/vite-plugin-gsx`. `npm run dev` *is* the full loop.
- `vite.config.ts` → `gsx({ watch: false })` (receives events; no daemon spawn). Proxy +
  `devFallback` + base stay; the file no longer owns generation.
- `main.go.tmpl` → drop `vite.NotifyReload`; plain HTTP server (keeps `vite.New()` asset
  resolution + graceful shutdown).

**Kept:** `app.gsx`, `.env` (ports), `web/main.js`, `web/style.css`, `public/*.svg`.

Net result the user sees: `go tool gsx dev`, a vanilla `main.go`, a vite config, and
`.env`. Minimal deps, small number of files, easy to understand.

## 6. Command shape & code layout

- `case "dev": return runDev(cmdArgs, stdout, stderr, cfg, workDir)` in `gen/main.go`'s
  dispatcher; usage line added to `printUsage()`.
- `runDev` lives in a new **`gen/dev.go`** (the orchestrator: flag/`[dev]` resolution, child
  supervision, the cycle loop, teardown). Process-group helpers (`Setpgid`, `killGroup`) go
  in a unix-tagged file (`gen/dev_unix.go`) so the Windows follow-up is a sibling build-tag
  file.
- Reuses `gen/watchsession.go` (`newWatchSession`, `regenDir`, reverse-closure) and the
  fsnotify setup from `gen/watch.go` for the watch + warm-regen half; adds `.env*` to the
  watched set and the server-rebuild/restart + browser-POST stages on top.

## 7. Testing

`gsx dev` is a tooling/runtime command (not syntax/codegen), so **no corpus case**. Instead:
- **Unit tests** — `[dev]` parsing + the flag → `gsx.toml` → default precedence chain;
  argv splitting (flag whitespace-split vs config array); the watch → action mapping
  (which change triggers regen-only vs rebuild vs env-restart).
- **Integration test** (style of `gen/init_test.go`) — build gsx, run `gsx dev` against a
  scaffold fixture under a **PTY**, send Ctrl-C, and assert: clean teardown (ports free, no
  orphan), restart-on-`.go`-change with vite untouched, `.env`-change restart, and
  `tmp/dev.log` contents (codegen + build + server stages present). Mirrors the prototype's
  validated `spike_test.py` harness.

## 8. Rollout / sequencing

1. **`vite-plugin-gsx`**: add the `watch: false` receive-events option (the plugin can keep
   its daemon path; the new option just adds an HTTP event sink reusing the existing
   overlay/reload logic). Ship first so gsx dev has something to talk to.
2. **`gsx dev`**: `gen/dev.go` + `gen/dev_unix.go`, dispatcher case, unit + integration
   tests.
3. **Scaffold**: migrate `simple/` (§5); update `gen/init_test.go`
   (`TestInitTaskfileParses` → a `gsx dev` smoke/teardown test) and any docs.
4. **Independent adversarial review** (builds throwaway probe programs) before merge, per
   project process.
5. **Follow-ups** (separate): Windows process-group teardown; `--watch`/`--exclude` flags if
   needed; `docs/guide` dev-loop page + `ROADMAP.md` update.
