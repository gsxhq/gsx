# Poison `.x.go` on generation failure ‚Äî design

**Date:** 2026-07-10
**Status:** Approved (brainstorm complete, pending implementation plan)

## Problem

When a `.gsx` file fails to parse or generate, gsx leaves the previously generated
`.x.go` untouched. Two consequences:

1. `go build` silently compiles and ships stale code.
2. Nothing on disk or in the build indicates the `.gsx` is broken ‚Äî the error is
   only visible if you happen to run `gsx generate` (or are inside `gsx dev` / the LSP).

It is worse than it looks: the write gate is per **package directory**
(`gen/cache.go:166-170`, and again in `writeAll` at `:255-258`) ‚Äî one broken
`.gsx` suppresses fresh output for every sibling in the package, so all of them
go stale at once. The generated header
(`// Code generated by gsx; DO NOT EDIT.`, `internal/codegen/emit.go:336-349`)
carries no source name or hash, so staleness is undetectable from the file itself.

## Prior art

- **templ** (closest analog) leaves the stale `_templ.go`, relies on exit codes,
  ships `templ generate -check` for CI, and documents commit-generated-code +
  `git diff --exit-code`. A proposal to validate generated output (templ #643)
  was closed "not planned".
- **sqlc** behaves the same: generate in memory, abort before writing on error.
- **gqlgen** is the cautionary tale: accidental delete-on-failure leaves projects
  uncompilable (#2345), and loading its own generated output creates sticky
  chicken-and-egg failures (#112).
- **stringer** and **protoc-gen-go** deliberately emit code that fails compilation
  as a signal (the `_ = x[FooA-0]` tripwire; `protoimpl.EnforceVersion`).
  Deliberate compile errors as messaging are mainstream Go practice.
- **cgo / goyacc** use `//line` directives so compiler errors point at the
  original source.
- **Rust build scripts / C# source generators / Vite transforms** make generation
  part of the build, so staleness is structurally impossible. Go offers no such
  hook ‚Äî `go generate`'s decoupling from `go build` is an explicit design decision.

No shipped tool writes a deliberately non-compiling file on generation failure,
but every ingredient of doing so is precedented.

## Options considered

1. **templ parity** ‚Äî `gsx generate -check` + drift docs. Doesn't fix local
   `go build`; rejected as the core fix (a `-check` flag remains a possible
   follow-up nicety).
2. **Delete the `.x.go` on failure** ‚Äî build breaks with baffling
   `undefined: MyComponent` errors at call sites; destroys last-good output for
   nothing in return. Rejected.
3. **Poison the `.x.go` on failure** ‚Äî overwrite with a deliberately
   non-compiling file carrying the real error. **Chosen.**
4. **Hash-stamp header + staleness check** ‚Äî complements 1/3 but doesn't fix
   `go build` by itself. Not in scope.

## Invariant

> A package whose gsx generation failed can never `go build` successfully, and
> the build error points at the broken `.gsx` with its real error message.

## Design

### Poison file format

Deterministic bytes (no timestamps), so re-running generate on the same failure
is byte-identical and the existing byte-equality skip in `restore`
(`gen/cache.go:207-209`) prevents write churn:

```go
// Code generated by gsx; DO NOT EDIT.
package ui

// GSX GENERATION FAILED for app.gsx. Fix the errors below and re-run gsx generate.
//
//	app.gsx:12:5: unexpected token '<' in attribute value
//	app.gsx:30:2: undefined filter "upper"

//line app.gsx:12:5
var _ = GSX_GENERATION_FAILED__see_app_gsx
```

- All error diagnostics for the file listed in the comment block, in the
  already-sorted diagnostic order.
- One `//line` directive pointing at the **first** error position, so `go build`
  prints `app.gsx:12:5: undefined: GSX_GENERATION_FAILED__see_app_gsx`.
- Syntactically valid Go ‚Äî the undefined identifier is a *type* error ‚Äî so
  go/parser, the skeleton scanner, gofmt, and editors handle the file without
  choking; only compilation fails.
- Standard gsx header, so the file is recognized as gsx-owned and freely
  overwritten on the next successful generate.
- Same `//go:build` pass-through directives as a normal emit, so build-tag
  variants stay consistent.
- The identifier embeds a sanitized form of the source filename
  (`GSX_GENERATION_FAILED__see_app_gsx`) so the one-line compiler error is
  self-explanatory.
- **Implementation-time check:** the compiler uses `//line` filenames verbatim
  in error output. Pick the form that is clickable in editors (likely the path
  relative to the package dir, where `app.gsx` actually sits) and pin it in a test.

### When poison is written

- **Per-file blame:** every `.gsx` with at least one error-severity diagnostic
  attributed to it gets its `.x.go` poisoned ‚Äî including when no `.x.go` existed
  yet (a brand-new broken `.gsx` produces a clear build error instead of
  `undefined: MyComponent` noise at call sites).
- **Clean siblings in a failing package:** left as-is (stale). Safe *because of
  the invariant* ‚Äî the poisoned sibling makes the whole package unbuildable, so
  stale siblings can't be silently used. This avoids restructuring
  `module_importer.go`'s abort-the-whole-package flow (`:749-760`) in this change.
- **Package-level errors with no attributable file** (rare ‚Äî e.g. synthetic
  diagnostics carrying no filename): conservatively poison every gsx-owned
  `.x.go` in that package dir, each carrying the package-level error text and no
  `//line`. The invariant must hold even when blame is fuzzy.
- **Operational errors** (I/O, module-graph failures where generation didn't
  meaningfully run): no poisoning ‚Äî exit 1 as today. Poison only when gsx
  positively determined "this source is broken".
- **`gsx dev`:** identical behavior ‚Äî poisoning is unconditional. No flags, no
  config.

### Write-path changes

- The per-dir gate at `gen/cache.go:166-170` / `:255-258` ("any error in dir ‚Üí
  write nothing") becomes: write poison for blamed files, skip the rest. Poison
  goes through the same `restore` path (byte-equality skip included), so a later
  successful generate overwrites it via ordinary byte inequality.
- `restore`'s plain `os.WriteFile` (`cache.go:211`) becomes temp+rename in the
  same directory. Writing intentionally-broken files makes atomicity actually
  matter: a *truncated* poison file is a parse error that could confuse the LSP
  and skeleton scanner, which the format above deliberately avoids.

### `gsx dev` implications

- **No watcher feedback loop.** `watchable()` hard-excludes `.x.go`
  (`gen/watch.go:220-222`, "never react to our own output") and `sourceTracker`
  won't snapshot them, so poison writes never retrigger a regen cycle.
- **Mid-session broken edit: UX unchanged.** A failed cycle `continue`s before
  the build step (`gen/dev.go:250-252`) ‚Äî the last-good *binary* keeps serving
  and the overlay shows the error, exactly as today. The poison lands on disk
  but `gsx dev` never compiles it. The visible difference is outside the loop:
  a parallel `go build` / `go test` in another terminal now fails truthfully.
- **Startup on a broken tree ‚Äî required tweak + accepted behavior change.**
  Today `dev.go:118` runs the initial build unconditionally even when the cold
  generate failed; stale `.x.go` compiles and the server runs old code beneath
  the overlay. Under poisoning that build fails, and the `buildErrorEvent`
  (`dev.go:119`) would replace the rich gsx overlay with a raw compiler error.
  **Tweak: skip the initial build when `startOK` is false** (mirrors the watch
  loop). Consequence, accepted: starting `gsx dev` on a broken tree serves no
  backend until the error is fixed (overlay only). There is no soft option ‚Äî
  the cold generate has already poisoned the file, so no last-good `.x.go`
  remains to build.
- **Recovery.** Fixing the `.gsx` always changes bytes (poison ‚Üí good), so
  `wrote == true` and the normal rebuild+reload path fires. The `overlayUp`
  machinery (`dev.go:124-128`, `:257`) stays ‚Äî still needed for build-error and
  `.env` recovery ‚Äî but gsx-error recovery no longer depends on it. Cost: when
  the fix restores output identical to the pre-error state, one redundant
  rebuild (near-free via the go build cache).
- **Editor/gopls while broken: truthfully red instead of silently green.**
  gopls reports the generated package broken and cross-package call sites show
  `undefined`. Noisier mid-edit (autosave), but it's the same truth `go build`
  now tells; the gsx LSP still points at the actual `.gsx` error.
- **Plumbing note.** `regenPending`'s `r.Written` becomes non-empty on failed
  cycles (the poison writes). The loop's early `continue` means nothing misreads
  it today; watch-session tests must pin that failed cycles report poison writes
  distinctly, so a future refactor doesn't conflate "wrote poison" with "wrote
  good output".

### Self-reference safety (the gqlgen trap)

Generation, the LSP, and the attrs-only probes all load packages containing
`.x.go` files. A poison file must never make failure sticky. The
skeleton-overlay architecture should already shield this ‚Äî real `.x.go` files
whose `.gsx` is present are replaced by synthetic skeletons in the overlay (the
key-presence check at `internal/codegen/module_importer.go:964`) ‚Äî but this gets
**deliberate regression tests**, not an assumption:

- poison a file ‚Üí fix the source ‚Üí re-run generate ‚Üí assert clean output;
- the same cycle through the LSP analysis path;
- an attrs-only probe scenario with a poison file present.

### What doesn't change

Dev overlay, LSP diagnostics, CLI rendering and exit codes ‚Äî all as today. This
change only alters what lands on disk. No new flags, no config.

## Testing

- Unit/integration tests in `gen`:
  - broken file ‚Üí exact poison bytes pinned (golden);
  - fix ‚Üí clean regeneration;
  - idempotence (second failing run writes nothing);
  - new-file case (no pre-existing `.x.go`);
  - no-filename package-level case (all `.x.go` in dir poisoned);
  - sibling-left-stale-but-package-unbuildable: actually run `go build` in the
    test to pin the invariant and the `//line`-redirected error text.
- Self-reference cycle tests (above).
- `gsx dev` watch-session tests: failed cycle writes poison + skips build;
  startup-on-broken-tree skips initial build and keeps the gsx overlay;
  recovery rebuilds and reloads.
- Corpus stays untouched ‚Äî it pins successful generation; failure-shape is
  `gen`-package territory.

## Documentation

A short guide section ("what happens when generation fails") ‚Äî users will see
poison files in git diffs, and committing one is a feature: CI fails loudly.
