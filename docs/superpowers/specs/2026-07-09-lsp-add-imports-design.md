# LSP add-import: `source.organizeImports` + `quickfix`, without a filesystem scan

## Problem

`gsx`'s language server removes and reorders imports, but never **adds** one. On
`learn-gsx-init/gsx-app/test.gsx`:

```gsx
package main

var hello = "Hello, World!"

var xx = <p>{ fmt.Sprintf(hello) }</p>
```

the server reports `undefined: fmt` and offers **zero** code actions. Verified by
driving the real LSP over stdio: `organizeImports actions: 0`, `ALL code actions offered: 0`.

gopls, by contrast, both folds add-import into `source.organizeImports` and offers an
`Add import: fmt` quickfix on the diagnostic.

## Why we cannot just call goimports

The public surface of `golang.org/x/tools/imports` is:

```go
type Options struct { Fragment, AllErrors, Comments, TabIndent, TabWidth, FormatOnly bool }
func Process(filename string, src []byte, opt *Options) ([]byte, error)
```

There is **no resolver hook**. The type that would accept one, `ProcessEnv`, lives in
`golang.org/x/tools/internal/imports` — `use of internal package … not allowed`.

So `imports.Process` with `FormatOnly: false` means accepting goimports' own resolver,
which walks the module cache and GOPATH. Measured against the real `gsx-app` module:

| case | cold | warm | result |
|---|---|---|---|
| missing stdlib (`fmt`) | 7.0 ms | 6.8 ms | adds `import "fmt"` |
| missing third-party (`gsx`) | 7.0 ms | 5.9 ms | adds `github.com/gsxhq/gsx` |
| **unresolvable** (`zzznotapkg`) | **1.41 s** | **1.32 s** | adds nothing |

An unresolved identifier is the *normal* mid-typing state, and `imports.Process` keeps no
cache across calls. This directly contradicts the LSP-performance rule enforced in PR #64
(no `packages.Load`/`go list` on an analysis path) and main's `CLAUDE.md` note that
`packages.Load` is expensive.

**We do not need it.** goimports must *guess* which package provides `fmt` by scanning the
filesystem. gsx has strictly better inputs: a type-checked skeleton, and a map of every
package already in the module's dependency graph.

## Detection: type-driven, not error-string parsing

`analyze` type-checks a skeleton that contains the user's `fmt.Sprintf(hello)`. A missing
qualifier is exactly a `*ast.SelectorExpr` whose root `*ast.Ident` has **no entry in either
`info.Uses` or `info.Defs`**. Verified:

```
local    .Field       Uses=true  Defs=false  local/var: var local struct{Field int}
strings  .ToUpper     Uses=true  Defs=false  PkgName (imported)
fmt      .Sprintf     Uses=false Defs=false  *** NO OBJECT -> undefined qualifier ***
rand     .IntN        Uses=false Defs=false  *** NO OBJECT -> undefined qualifier ***
```

Locals resolve to a `var`; imported packages to a `PkgName`; only genuinely undefined
qualifiers have no object. No parsing of `"undefined: fmt"` message text.

The **selector name** (`Sprintf`) is captured alongside the qualifier. It is what makes
disambiguation possible at all.

## Resolution: two maps, no scan

`resolveImportCandidates(name, symbol) []string` unions two sources.

1. **Dependency graph.** `externalImporter` loads with
   `NeedName|NeedTypes|NeedImports|NeedDeps` into a `mapImporter`, which is a plain
   `map[string]*types.Package` — enumerable. Every package in the module's transitive graph
   is there, with its real declared `Name()` and a populated `Scope()`. Gate on
   `Complete()` (per PR #64: `go/types` fabricates a placeholder named after the path base
   for anything its importer never loaded — never trust `Name()` on an incomplete package).
   Covers `github.com/gsxhq/gsx`, the project's own packages, sibling `.gsx` packages whose
   `.x.go` exists, and every stdlib package already reachable (`fmt`, `context`, `io`).

2. **Baked stdlib table.** `name -> []path` over the 193 non-`internal`, non-`vendor` std
   packages, generated by `go:generate` from `go list std`. Covers std packages absent from
   the graph (`text/tabwriter`, `container/ring`). A test diffs the table against a live
   `go list std` so a Go upgrade that adds or renames a package fails loudly rather than
   silently under-resolving. (Skipped under `-short`.)

### Disambiguation by symbol

Only ~6 std names collide meaningfully (`rand`, `template`, `scanner`, `pprof`, `json`,
`asn1` — `internal`/`vendor` excluded). When the union has more than one candidate, keep
only packages that actually export `symbol`:

- **Candidate already in the graph** → `pkg.Scope().Lookup(symbol) != nil`. Free.
- **Candidate only in the stdlib table** → load its export data via
  `importer.ForCompiler(fset, "gc", nil)`. Measured ~30–50 ms cold, ~25 µs warm (the
  importer caches). Verified to disambiguate correctly:

  ```
  math/rand    exports IntN = false     math/rand/v2  exports IntN = true
  text/template exports HTML = false    html/template exports HTML = true
  ```

An importer failure drops that candidate (conservative: offer fewer, never add a wrong
import). In `Bundle`/WASM mode there is no `GOROOT` export data; table-only candidates are
then simply not symbol-checked, so an ambiguous name yields multiple quickfixes and no
automatic add. Graceful degradation, not an error.

**Nothing here scans the filesystem.** An unknown name resolves to the empty slice and we
offer nothing — the user runs `go get`, which they would have had to do anyway.

## Splicing the import into the `.gsx`

`ast.File` is `{Doc, Package string, Decls []Decl}` — the package clause is not a decl, and
the parser peels a leading run of imports into a single `GoChunk`.

`gsxfmt.FormatOptions` gains `Add []ImportRef`:

- If the file has a leading import `GoChunk`, insert the new spec text at its top.
- Otherwise prepend a new `GoChunk{Src: "import \"fmt\""}` as `Decls[0]` — the `test.gsx`
  case, which has no imports at all. It must be `Decls[0]`: an `import` after a `var` is
  invalid Go.

Then the existing `reorderImports` (goimports `FormatOnly`) merges, dedups, groups, and
sorts. No new formatting machinery, and adds compose with removes in one edit.

## Hot-path contract

Detection is a walk over skeleton ASTs `analyze` already parsed: no IO, no lock, no
`packages.Load`, no importer call. `PackageResult.MissingImports` carries
`{Name, Symbol, Pos}` per `.gsx` path, **deliberately unresolved**. `Pos` is the
qualifier's `.gsx` position (mapped through the skeleton's `//line` directives, as the
existing diagnostics already are); the quickfix handler uses it to match the request's
`context.diagnostics` to a specific missing qualifier, and the same qualifier may legitimately
appear more than once in a file.

Resolution — including every export-data read — happens only inside a `Module` method that
the LSP calls from a **code-action handler**, which is user-triggered. It is never reachable
from `Package()`.

Guarded the same way PR #64 guards `packages.Load`: a counter on the resolution entry point,
asserted **0** across `Package()` and **non-zero** after a code-action resolve, so the zero
assertion cannot be vacuous.

## Surfaces

- **`source.organizeImports`** additionally adds every missing qualifier that resolves to
  **exactly one** candidate. Ambiguous names are left undefined — never guessed. This runs
  non-interactively (format-on-save), so a wrong guess would be unrecoverable.
- **`quickfix`** on the `undefined: X` diagnostic: one action per surviving candidate —
  `Add import: math/rand/v2`, `Add import: crypto/rand`. Requires advertising `quickfix` in
  `codeActionKinds` next to `source.organizeImports`, and matching the request's
  `context.diagnostics` to the missing qualifier by position.

Both edits reuse the existing whole-document `TextEdit` path.

## Components

| Unit | Responsibility |
|---|---|
| `internal/codegen` (analyze) | Detect missing qualifiers from skeleton + `types.Info`. Hot path. |
| `internal/codegen` (resolver) | `stdlibIndex` (baked) + `depGraphIndex` (from `mapImporter`); symbol disambiguation. Off hot path. |
| `internal/codegen/stdlibindex.go` | Generated `name -> []path` table + `go:generate` directive. |
| `internal/gsxfmt` | `FormatOptions.Add`; splice into or create the leading import chunk. |
| `internal/lsp` | `quickfix` handler; `organizeImports` gains unambiguous adds. |
| `gen/lsp.go` | Implement the new `Analyzer` resolve method on `lspAnalyzer`. |

## Non-goals

- **No CLI add.** `gsx fmt` keeps its type-check-free syntactic path (remove + reorder).
  Adding needs type information; `gsx fmt -l .` is a CI drift gate and must stay fast.
- **No module-cache scan.** A package not in `go.mod` is not offered. Reaching goimports
  parity there requires a background-refreshed index (what gopls maintains) and is a
  separate project.
- **No `-local` grouping**, no change to `gofmt`/`goimports` mode selection.

## Testing

- **Detection:** local vars, imported packages, dot imports, shadowed names, method values,
  generic type args, and qualifiers inside `{ }` interpolation / attributes / `<script>` /
  `{ if }` must **not** be reported missing. A qualifier used only in one of those positions
  and undefined **must** be reported.
- **Resolution:** `fmt` → `["fmt"]`; `rand` + `IntN` → `["math/rand/v2"]` only; `template` +
  `HTML` → `["html/template"]` only; `rand` + a symbol none exports → all candidates, so
  organizeImports adds nothing and two quickfixes appear; unknown name → `[]`.
- **Incomplete-package guard:** an out-of-graph package must never contribute its fabricated
  path-base name (regression guard for the PR #64 Critical).
- **Hot-path counter:** 0 resolution calls from `Package()`; non-zero from the resolve
  method.
- **Stdlib table freshness** against live `go list std`.
- **Splicing:** file with no imports (prepend), file with a leading import block (splice +
  merge), file where the added import duplicates an existing one (dedup), add + remove in
  one edit.
- **End-to-end:** drive the real LSP on `test.gsx`; assert `undefined: fmt` yields both a
  quickfix and an `organizeImports` edit producing `import "fmt"`.
