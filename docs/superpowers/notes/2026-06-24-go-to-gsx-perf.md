# gsx LSP .go→.gsx: Performance Baseline (2026-06-24)

## Fixture

Generated by `gen/perf_test.go` (`TestPerfBaseline`, requires `GSX_PERF=1`).

| Dimension | Value |
|-----------|-------|
| Packages (N) | 50 |
| `.gsx` files | 200 (4 per package) |
| `.go` files | 50 (1 per package, referencing all 4 components) |
| Module layout | temp dir / single `go.mod` with `replace` → repo root |
| Platform | darwin arm64 (Apple M-series) |
| Go version | 1.26.1 |

Each package is a subdirectory of the synthetic module. Each `Analyze` call runs
`codegen.GeneratePackagesWithFilters` on that one directory (one `go/packages.Load`
per package, as the LSP server does today).

---

## Measured Numbers

### `Analyze` latency (cold — first call per package)

| Stat | Value |
|------|-------|
| min | 328 ms |
| median | 383 ms |
| max | 518 ms |
| total (50 pkgs) | 19.5 s |

### `Analyze` latency (warm — second call, same packages, same process)

| Stat | Value |
|------|-------|
| min | 342 ms |
| median | 386 ms |
| max | 675 ms |

Cold ≈ warm: there is no meaningful caching between `Analyze` calls. Each call
pays the full `go/packages.Load` cost.

### Definition / references lookup latency (cross-index)

A real `CrossIndex` map lookup by component key, averaged over 100k iterations,
measures **well under 1 µs** (the handlers then range the entry's `Refs`, also
O(refs)). As expected, the cross-index query path is not a concern.

### Retained memory (heap, holding all 50 `*lsp.Package` results)

| Metric | Value |
|--------|-------|
| Heap delta (HeapInuse before→after) | 1,234 MiB |
| Per-package (heap) | **24.7 MiB** |
| Total alloc (sum across all Analyze calls) | 16,926 MiB |

The total-alloc figure confirms that `go/packages.Load` allocates heavily and
discards most of it via GC — but the 24.7 MiB per-package figure is the
**steady-state retained cost** of holding a `*lsp.Package` (Fset, TypesInfo,
ExprMap, gsx AST, CrossIndex).

### Slim cross-index size vs retained heap

| Metric | Value |
|--------|-------|
| CrossIndex entries | 200 (4 per package × 50) |
| `CrossRef` header size (`unsafe.Sizeof`) | 80 bytes |
| Slim index (incl. string + Refs bodies) | ~30–40 KiB |
| Slim index as % of retained heap | ~0.003% |

(The 80 B is the struct header only; counting each `Name` string body and the
`[]token.Position` backing arrays, the true per-entry cost is ~150–200 B, so the
whole 200-entry index is on the order of tens of KiB — still negligible.)

The cross-index is negligible. Retained memory is dominated by `types.Info`
(the full type-checker output, retained for `.gsx`-side go-to-definition) and
the two `token.FileSet`s.

---

## Dominant Costs

1. **`go/packages.Load` latency (~383 ms/package, no warm-up benefit)**
   This is the irreducible cost of type-checking one Go package via the
   standard toolchain. On a 50-package project, a naive "analyze every package
   on open" strategy would take ~19 s total at file-open time.

2. **Retained `types.Info` memory (~24.7 MiB/package)**
   The full `TypesInfo` (all Uses, Defs, Types maps) is retained per package
   because `.gsx`-side go-to-definition needs it for expression → position
   resolution. For 50 open packages this is ~1.2 GiB.

3. **`go/packages.Load` shows no warm-up gain**
   Cold ≈ warm latency confirms that `go/packages.Load` does not benefit from
   an in-process cache in the current architecture. Each `Analyze` call is
   independent (separate invocation of the Go type-checker).

---

## Conclusion and Optimization Recommendation

### Is any optimization warranted now?

**Yes, at scale.** A project with 50 packages where the developer opens files
across many packages will eventually hold ~1.2 GiB of retained `types.Info`
and will experience ~383 ms latency per new package opened. These numbers are
not catastrophic for a small/medium project (< 10 open packages ≈ < 250 MiB,
< 4 s on first open), but they become user-visible pain on larger codebases.

The `.go→.gsx` cross-index itself is negligible and does not need optimization.

### Recommended candidates (in priority order)

**1. Debounce `Analyze` calls (highest leverage, lowest risk)**
Rather than re-running `Analyze` on every keystroke / `didChange`, debounce
with a 300–500 ms quiet window. This is a behavioral change (editor
responsiveness), not a memory change, but eliminates redundant `Load` calls
during rapid edits. This is the standard LSP server pattern and should be the
first thing implemented.

**2. LRU cap on `s.pkgs` (the retained package map)**
Cap the number of `*Package` entries held in `Server.pkgs` (e.g., most-recently
used 10–20 packages). At 24.7 MiB per package, 10 packages = ~247 MiB — a
reasonable resident set for a background LSP server. Evicted packages are
re-analyzed on demand (paying the 383 ms cost again). This is a simple
`map[string]*Package` → bounded LRU map substitution with well-known Go
implementations.

**3. Slim the `.gsx`-side full-`Info` retention (medium complexity)**
`types.Info` is retained for `.gsx`-side expression→position resolution.
A smaller retained structure (e.g., only `Uses` and `Defs` maps, not `Types`
and `Implicits`) could cut per-package memory significantly, but requires
auditing which `TypesInfo` fields the LSP handlers actually read. Deferred
pending profiling (the 24.7 MiB includes the gsx AST and FileSets too, so
`Info` alone may be < half).

**4. Lazy `.go` attach / deferred cross-index**
Not warranted. The CrossIndex build cost is negligible (< 1 ms) and the slim
index is on the order of tens of KiB for the entire 50-package fixture. The
`.go→.gsx` design's "build once, discard inputs" intent is already correct.

### What NOT to do yet

Do not attempt to cache or share `go/packages.Load` results across packages.
That requires a persistent `go/packages` driver or `gopls`-style incremental
loading, which is a substantial architectural change. Debounce + LRU cap
addresses the practical pain first.
